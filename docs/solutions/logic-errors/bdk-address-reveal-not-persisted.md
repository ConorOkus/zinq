# BDK Address Reveals Not Persisted — Funds Invisible After Restart

## Problem

After a channel close or LDK sweep, the on-chain balance showed less than expected. Some UTXOs were invisible to the BDK wallet even though they existed on-chain. The balance only showed the most recent UTXO, missing earlier ones.

## Root Cause

The LDK event handler called `wallet.next_unused_address('external')` to get a sweep destination address, but **never persisted the BDK changeset** that marks that address as "revealed". BDK's `take_staged()` captures the address reveal, but `putChangeset()` was never called.

After a page reload, the wallet restored from the old changeset. Since the sweep address wasn't in the revealed set, `start_sync_with_revealed_spks()` didn't watch it. Funds at that address were invisible.

The same problem affected the custom `SignerProvider` which also called `next_unused_address()` for channel close destinations.

## Investigation

Traced via Mutinynet block explorer:

1. Found 3 UTXOs on-chain across different wallet addresses
2. Wallet only showed 1 UTXO (94,847 sats) instead of 3 (284,252 sats total)
3. The missing addresses were generated by `next_unused_address()` in the event handler
4. BDK changeset in IndexedDB had `last_revealed` with only index 0 for the external chain

## Solution

Two fixes applied together:

### 1. Persist changeset after every `next_unused_address()` call

```typescript
const addressInfo = bdkWallet.next_unused_address('external')
// Persist the address reveal so BDK syncs this address after restart
const staged = bdkWallet.take_staged()
if (staged && !staged.is_empty()) {
  void putChangeset(staged.to_json()).catch((err) =>
    console.warn('Failed to persist address reveal:', err)
  )
}
```

Applied in all 3 locations: `bdk-signer-provider.ts`, `event-handler.ts` (setBdkWallet), `event-handler.ts` (SpendableOutputs).

### 2. Always full-scan on wallet init

Changed `onchain/init.ts` to always use `wallet.start_full_scan()` instead of `start_sync_with_revealed_spks()` for restored wallets. This recovers any addresses that were revealed but not persisted.

```typescript
// Always full scan to discover all addresses, including any revealed
// by the LDK event handler but not persisted in the changeset
const fullScanRequest = wallet.start_full_scan()
const update = await esploraClient.full_scan(fullScanRequest, gapLimit, parallelRequests)
wallet.apply_update(update)
```

## Prevention

- **Every call to `next_unused_address()` must be followed by changeset persistence** — BDK advances the address index in memory but this is lost on restart without persistence
- `take_staged()` is destructive — it drains the changeset. If multiple call sites race, one may get an empty changeset. Call `take_staged()` + `putChangeset()` in the same synchronous block as `next_unused_address()`
- Always-full-scan on init is a safety net but should not be relied upon as the primary mechanism
